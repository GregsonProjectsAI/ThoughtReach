1. Template Axes
These are the variables the system needs in order to choose the right underlying game structure.

A. Player Count
This should stay exact, not ranged, at the product level.

Why:

role balance matters

artefact distribution matters

final puzzle participation matters

“one person standing around” kills perceived quality fast

For the matrix, though, we can still design around structural bands behind the scenes.

Use these structural bands internally:

P1: 2–3 players

P2: 4–6 players

P3: 7–10 players

Then later we create exact-count pack variants inside each band.

So:

template logic uses bands

production packs use exact counts

That keeps the system sane.

B. Environment Type
This is the most important structural split after player count.

Use two primary classes:

E1 — Contained
indoor

small garden

room-to-room

short walking distance

object-focused

discovery density is high

Typical use:

homes

flats

classrooms

party rooms

small venues

E2 — Expansive
outdoor

large house + garden

venue with multiple zones

longer travel between clue points

navigation matters more

pacing must account for movement time

Typical use:

gardens

parks

large event spaces

multi-floor houses

mixed indoor/outdoor settings

This split should stay hard.
Do not try to make one template do both.

C. Space Complexity
This is not “size” in a vague sense. It should mean:

How many distinct usable clue zones can the host reliably set up?

Use:

S1: 1 zone

S2: 2–3 zones

S3: 4–6 zones

S4: 7+ zones

A “zone” can be:

a room

a clearly distinct outdoor area

a staircase landing

a patio

a sofa area vs dining area if clearly separable

This is better than square footage because customers understand it faster and it maps to placement logic.

D. Game Length
Length should control:

number of tasks

number of transitions

artefact count

branch depth

final puzzle complexity

Use:

L1 Short: 15–25 min

L2 Standard: 30–45 min

L3 Extended: 45–60 min

That is enough for V1.

Do not go beyond 60 min in the initial system unless the product is specifically built for older kids/adults later.

E. Theme
Theme is not a structural axis.

It is a skin layer.

That means theme can change:

names

art style

mission wrapper

item aesthetics

narrative language

But it should not change:

clue count

puzzle type

flow order

branch logic

role balance

This is critical. If we let theme influence structure too early, the whole system becomes unstable.

2. What Actually Determines Template Selection?
Not every axis should create a new template.

If we combine all axes naively, we get too many templates too fast.

For V1, the true structural drivers are:

environment type

space complexity

length

player band

But some of those should affect variant rules inside a template, not create new templates.

So the hierarchy should be:

Primary structural selectors
These choose the template family:

environment type

space complexity

rough player band

Secondary modifiers
These tune the template:

game length

exact player count

Skin modifier
theme

That means:

we don’t need 40 templates

we need a few template families with controlled variants
